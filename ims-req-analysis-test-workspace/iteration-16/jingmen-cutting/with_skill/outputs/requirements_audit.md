## Critical Issues (block testing)

| ID | Issue | Location | Impact | Suggested Fix |
|----|-------|----------|--------|---------------|
| AUD-001 | CPO交期排序的次优先级未定义——当两个CPO交期相同时，自动充数模式优先选哪个？ | 4.1 自动充数模式 / 4.2 手动充数模式 | 自动充数逻辑不可确定性测试 | 明确次优先级：CPO编号升序 / 创建时间 / 订单数量 |
| AUD-002 | "智能合并"的合并算法未明确——"同交期"的判定精度是天级还是时分秒级？ | 1. 裁剪任务模块 | 合并结果不可预测，无法验证 | 明确定义交期相等判定粒度（建议天级） |

| AUD-004 | PC端超裁审核中"放行→增加计划层数"——增加的层数从哪来？是凭空增加还是从其他任务调配？ | 2.4 防超裁机制→PC端超裁审核 | 超裁放行后数据一致性不可验证 | 明确：增加唛架总层数 / 从其他任务调拨 / 独立超裁额度 |
| AUD-005 | 床次号"辅布F开头"的F前缀触发条件——什么情况下一个床次被判定为"辅布"？ | 2.3 床次号唯一性规则 | 床次号格式不可验证 | 明确辅布床次的判定依据：布卷类型 / 唛架类型 / 其他 |

## Assumptions Made

| ID | Assumption | Reason | Risk if Wrong |
|----|-----------|--------|---------------|
| ASM-001 | 交期排序使用日期比较（天级），同日CPO按CPO编号升序 | AUD-001/002未澄清前的默认规则 | 排序结果与产品预期不一致，自动充数测试需重做 |
| ASM-002 | 防超裁预警阈值10层为硬编码常量，非可配置项 | 需求文档未标注配置性 | 后续如需调整需代码变更 |
| ASM-003 | 超裁放行增加的是当前唛架的总计划层数 | 最常见实现方式 | 若为跨任务调拨，测试场景完全不同 |
| ASM-004 | 辅布床次判断基于拉布物料类型（辅料=辅布） | 合理推断 | 可能基于其他分类维度 |
| ASM-005 | 删除未开始的裁剪任务自动释放唛架已分配层数 | 资源管理基本逻辑 | 不释放会导致唛架层数泄漏 |
| ASM-006 | 床次号最大到9999，超出后循环或报错 | 三位数格式推断 | 大批量生产可能溢出 |
| ASM-007 | PAD端和PC端共享同一个后端会话，实时层数计算为服务端原子操作 | 防并发基础假设 | 若为客户端计算可能有竞态 |
