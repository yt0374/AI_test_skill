## 1. Overview + Business Context

### 1.1 项目背景

荆门鹰美原有裁剪系统对 CPO（Customer Purchase Order）采用强制分配策略：每个扎卡必须打印 CPO 号，缝制上线时严格按 CPO 组织生产。该模式在固定大订单、单一交期场景下运行尚可，但在实际生产中暴露两大问题：

1. **交期冲突**：CPO 交期临时提前时，原系统难以快速响应。
2. **现场适应性差**：内销单、多色号混合裁剪等场景不需要严格区分 CPO，强制绑定反而增加复杂度。

### 1.2 改造目标

从"强制分配到 CPO"转变为"按需绑定 CPO"，做到松紧有度：
- 需要精确追踪的订单保留 CPO 控制
- 不需要的场景可自由合并
- 适配工厂现场灵活多变的生产节奏

### 1.3 项目目标

| 目标 | 描述 |
|------|------|
| **交期优先** | 支持对特定 CPO 单独建裁剪任务、优先裁剪，保障交期 |
| **柔性作业** | 裁剪现场不预设固定床次，员工可实时创建床次并录入层数，系统自动防超裁 |
| **精准上线** | 缝制挂片站支持自动/手动匹配 CPO，确保生产与订单交期对齐 |

### 1.4 荆门客户差异

## 2. Functional Points

### F001: 裁剪任务创建 — 多CPO合并
用户创建裁剪任务时，可自由选择一个或多个 CPO，或按交期智能合并。系统基于所选 CPO 的订单数量生成裁剪任务。

### F002: 裁剪任务创建 — 全选 CPO
一键选中当前订单下所有 CPO，快速创建覆盖全部订单的裁剪任务。

### F003: 裁剪任务创建 — 智能合并（按交期）
系统根据交期智能合并 CPO：将同交期的 CPO 自动归为一组，用户确认后创建裁剪任务。

### F004: 唛架计划层数分配与递减
唛架作为公共资源可被多个裁剪任务引用。用户为 CPO 创建裁剪任务时，从唛架总计划层数中划拨层数，系统自动递减可用层数。

### F005: 唛架防超分配校验
分配给所有裁剪任务的计划层数之和不得超过唛架总计划层数，超出禁止提交。

### F006: 顶层裁剪任务管理（PC端）
PC端创建裁剪任务（顶层拉布任务），发布后 PAD 端同步可见。任务信息：关联CPO、计划层数、计划件数、生产组别。

### F007: 裁剪任务状态管理
未开始、拉布中、已完成三种状态。未开始时可编辑或删除；一旦有小组开始拉布（"拉布中"），编辑/删除按钮隐藏。

### F008: 床次创建（PAD端）
拉布员在 PAD 端进入裁剪任务后，点击【新增床次】，填写布卷号（扫码/手动）和拉布层数。

### F009: 并发防超检验
系统实时计算剩余可用层数 = 计划层数 - 已锁定层数 - 暂扣层数。输入层数 <= 剩余可用层数则通过；> 则触发防超裁。

### F010: 床次提交完成
拉布员点击【提交完成】，录入实际拉布层数。系统锁定实际层数，床次状态更新为"已拉布"。校验实际层数 <= 暂扣层数 + 实时剩余可用层数。

### F011: 床次号生成与唯一性
床次号在生产订单号内全局唯一，同一生产订单下多个裁剪任务共享同一递增序列。格式：主布C开头/辅布F开头 + 三位数字序号。在提交完成时生成。

### F012: 防超裁 — 软预警
全局剩余层数低于预设阈值（如10层）时，PAD端弹出提示："剩余层数不足X层，请注意控制。"

### F013: 防超裁 — 强控禁止
拉布员输入层数导致全局剩余层数为负时，系统禁止创建/提交，弹出："总裁剪层数已超计划，需PC端审核。"

### F014: 超裁审核（PC端）
超裁申请被触发后，PC端对应裁剪任务下出现【申请超裁】按钮。负责人审核后可放行（增加计划层数）或驳回。

### F015: 扎卡打印 — 合并CPO场景
裁剪任务包含多个 CPO 时，扎卡打印页面隐藏 CPO 选择区域，扎卡上不印刷 CPO 字段，按布卷号分扎。

### F016: 扎卡打印 — 单一CPO场景
裁剪任务仅包含一个 CPO 时，页面显示该 CPO 信息，扎卡上印刷 CPO 号，按 CPO 件数分扎。

### F017: 挂片站匹配 — 合并CPO
通用扎卡到达挂片站时，系统按交期优先顺序自动将裁片分配到 CPO。

### F018: 挂片站匹配 — 单一CPO
指定扎卡到达挂片站时，系统优先绑定指定 CPO；满额后自动按交期顺延到其他 CPO。

### F019: 自动充数模式（默认）
系统根据生产订单下所有 CPO 的交期顺序（最紧急优先），自动充数。指定扎卡优先绑定指定 CPO，满额后顺延。

### F020: 手动充数模式
员工手动切换至手动模式，自主选择 CPO 挂片。CPO 满额时弹出提示"CPO [编号] 数量已满，请切换其他 CPO"。手动模式下可跳过交期顺序自由选择。

### F021: 模式切换联动
自动→手动：员工随时可切换，保存当前进度。手动→自动：点击【恢复自动充数】，系统按实时交期重新计算目标 CPO，不参考手动切换过的 CPO。

## 3. Business Rules

| ID | 规则 | 来源 |
|----|------|------|
| R001 | 裁剪任务仅对所选 CPO 的订单数量创建，未选 CPO 的订单不参与 | F001 |
| R002 | 唛架已分配层数 = 所有裁剪任务（含已创建+执行中）分配层数之和 | F004 |
| R003 | 唛架剩余可用层数 = 总计划层数 - 已分配层数，<= 0 时禁止新分配 | F005 |
| R004 | 剩余可用层数（床次） = 计划层数 - 已锁定层数(sum) - 暂扣层数(sum) | F009 |
| R005 | 床次创建时：输入层数 <= 剩余可用层数 → 通过；> → 触发防超裁 | F009 |
| R006 | 提交完成时：实际层数 <= 暂扣层数 + 实时剩余可用层数 → 通过 | F010 |
| R007 | 床次号作用域：生产订单号内全局唯一，同一生产订单下跨裁剪任务共享递增序列 | F011 |
| R008 | 床次号生成时机：拉布员点击【提交完成】时，非创建时 | F011 |
| R009 | 床次号并发：同一秒内多次提交以数据库记录ID为准，确保严格递增 | F011 |
| R010 | 裁剪任务"未开始"可编辑/删除；"拉布中"隐藏编辑/删除按钮 | F007 |
| R011 | 软预警阈值：剩余层数 < X 层触发警告（X 可配置，默认10） | F012 |
| R012 | 强控阈值：剩余层数 < 0 时禁止提交，提示"总裁剪层数已超计划" | F013 |
| R013 | 超裁申请仅在强控触发后出现，PC端负责人审核放行/驳回 | F014 |
| R014 | 扎卡CPO显示规则：合并CPO→隐藏；单一CPO→印刷 | F015/F016 |
| R015 | 挂片自动充数规则：通用扎卡按交期优先，指定扎卡优先绑定+满额顺延 | F019 |
| R016 | 手动充数规则：员工可自由选择任意未满额CPO，无交期限制 | F020 |
| R017 | CPO满额后该CPO不可再挂片，系统弹出提示强制切换 | F020 |
| R018 | 分扎规则：合并CPO按布卷号分扎；单一CPO按CPO件数分扎 | F015/F016 |
| R019 | 手动→自动切换时，系统按实时交期重新计算，不参考手动选择的CPO | F021 |

## 4. Data Flow

### 4.1 整体数据流

```
销售订单 (已存在，荆门可能通过ERP同步)
  │
  ├── CPO1 (交期7/1, 500件)
  ├── CPO2 (交期7/5, 300件)
  └── CPO3 (交期7/2, 200件)
       │
       ▼
PC端: 裁剪任务创建
  ├── 选择CPO（单选/多选/全选/智能合并）
  ├── 分配唛架计划层数（防超分配校验）
  └── 发布 → PAD端同步可见
       │
       ▼
PAD端: 床次任务（现场拉布）
  ├── 新增床次 → 布卷号 + 层数 → 防超检验 → 暂扣层数
  ├── 提交完成 → 实际层数 → 锁定层数 → 生成床次号
  └── 防超裁：预警(剩余<N) / 强控(剩余<0) / PC端超裁审核
       │
       ▼
扎卡打印
  ├── 合并CPO → 隐藏CPO → 通用扎卡（按布卷号分扎）
  └── 单一CPO → 显示CPO → 指定扎卡（按CPO件数分扎）
       │
       ▼
缝制挂片站 (PAD)
  ├── 自动充数：按交期优先 → CPO满额切换
  └── 手动充数：员工自选CPO → 满额提示切换
```

### 4.2 层数流转示意

```
唛架总计划层数: 100
  │
  ├── 裁剪任务A(CPO1): 分配40层 → 占用40 → 剩余60
  ├── 裁剪任务B(CPO2+CPO3): 分配50层 → 占用50 → 剩余10
  │
  ▼
裁剪任务A (计划40层)
  ├── 床次001(小组1): 创建暂扣15层 → 提交锁定15层 → 剩余25
  ├── 床次002(小组2): 创建暂扣10层 → 提交锁定10层 → 剩余15
  └── 床次003(小组1): 创建暂扣15层 → 提交锁定15层 → 剩余0
       → 任务状态: 已完成
```

## 6. Data Entities & Variations

### 6.1 核心实体

| 实体 | 关键属性 | 来源 |
|------|----------|------|
| CPO | CPO号、交期、订单数量 | 生产订单 |
| 唛架 | 唛架编号、总计划层数、款号 | 数据-款号管理 |
| 裁剪任务 | 关联CPO、计划层数、计划件数、生产组别、状态 | 生产-裁剪任务 |
| 床次 | 床次号、布卷号、拉布层数、实际层数、状态、小组 | 生产-床次任务 |
| 扎卡 | 扎卡类型(通用/指定)、CPO号、布卷号、层数 | 生产-扎卡打印 |
| 生产订单 | 生产订单号、款号、CPO列表 | 生产-生产订单 |

### 6.2 状态机

**裁剪任务状态机：**
```
PC端创建 ──→ 未开始 ──[有小组开始拉布]──→ 拉布中 ──[所有床次完成+剩余层数=0]──→ 已完成
   │              │
   │         [可编辑/删除]          [编辑/删除按钮隐藏]
   │
   └── 发布 ──→ PAD端同步可见
```

**床次状态机：**
```
[新增床次] ──→ 拉布中（暂扣层数）──[提交完成]──→ 已拉布（锁定层数）
                    │
                    └── [输入层数 > 剩余] ──→ 触发防超裁
```

### 6.3 数据变量维度

| 维度 | 变量 | 影响测试的差异 |
|------|------|----------------|
| CPO数量 | 0个、1个、多个、全部 | 裁剪任务创建规则、扎卡类型、挂片策略 |
| CPO交期 | 同交期、不同交期、紧急交期 | 智能合并、自动充数优先级 |
| 唛架可用层数 | 充足、不足、刚好、为0 | 分配通过/防超分配 |
| 床次剩余层数 | 充足、低于阈值、为0、为负 | 通过/预警/强控 |
| 实际层数 vs 暂扣层数 | 等于、小于、大于可调整范围 | 提交通过/失败 |
| 扎卡类型 | 通用扎卡、指定扎卡 | CPO显示、挂片策略 |
| 挂片模式 | 自动充数、手动充数 | 分配规则、CPO选择 |
| 并发场景 | 单小组、多小组同时、同一秒提交 | 床次号唯一性、防超检验 |

## 7. Edge Cases & Constraints

### 7.1 边界场景

| ID | 场景 | 预期行为 |
|----|------|----------|
| E001 | 裁剪任务选择0个CPO | 系统阻止创建，提示至少选择一个CPO |
| E002 | 唛架剩余层数=0时分配新任务 | 禁止提交，提示层数不足 |
| E003 | 床次剩余层数=0时新增床次 | 禁止创建，提示剩余层数不足 |
| E004 | 床次剩余层数刚好等于输入层数 | 通过，剩余变为0 |
| E005 | 同时多个小组对同一裁剪任务新增床次 | 防超检验应基于实时数据，防止超额 |
| E006 | 同一秒内多个床次提交完成 | 床次号严格递增，以数据库记录ID为准 |
| E007 | 手动模式下CPO满额后不切换CPO继续挂片 | 禁止挂片到满额CPO，强制提示 |
| E008 | 裁剪任务"拉布中"状态尝试编辑/删除 | 按钮隐藏，操作不可用 |
| E009 | 扎卡打印页面打开合并CPO的裁剪任务 | 不显示CPO选择区域 |
| E010 | 指定扎卡的CPO在实际生产中已满额 | 自动顺延到其他CPO（交期优先） |
| E011 | 超裁申请被驳回后再次尝试 | 需再次申请，系统不记忆前次驳回 |
| E012 | 手动→自动切换时，所有CPO均已满额 | 系统需处理此状态（待澄清） |
| E013 | 裁剪任务创建时未分配生产组别 | 是否允许？后续床次能否创建？（待澄清） |

### 7.2 约束条件

| ID | 约束 |
|----|------|
| C001 | 裁剪任务仅在"未开始"状态可编辑/删除 |
| C002 | 床次号作用域为一个生产订单号内 |
| C003 | 主布床次号C开头，辅布床次号F开头 |
| C004 | 合并CPO时扎卡不显示CPO信息 |
| C005 | 单一CPO时扎卡强制显示CPO号 |
| C006 | 手动充数只能选择未满额的CPO |

## 8. API / Interface Notes

本文档未提供 API 接口规范。涉及的接口交互点：
- PC端与PAD端数据同步（裁剪任务发布后PAD可见）
- 床次创建/提交时的并发校验（需服务端实时计算剩余层数）
- 挂片站CPO充数逻辑（需服务端维护CPO满额状态）

**建议向开发团队确认：**
1. PC-PAD同步的接口协议和数据格式
2. 床次防超裁的后端计算逻辑
3. 挂片充数的CPO状态同步机制
