## 一、项目背景分析

### 1.1 业务痛点
原有系统采用CPO强制分配策略，存在以下核心问题：
- **交期冲突无法快速响应**：一个CPO交期临时提前时，原系统无法快速调整
- **现场适应性差**：内销单、多色号混合裁剪等场景不需要严格区分CPO，强制绑定增加了生产复杂度

### 1.2 改造策略
从"强制分配到CPO"转变为"按需绑定CPO"，实现松紧有度的管理模式：
- 需要精确追踪的订单 → 保留CPO控制
- 不需要的场景 → 自由合并

### 2.2 模块详细分析

#### 模块1：裁剪任务模块

**核心功能**：
1. CPO选择与合并
   - 勾选方式：单选或多选CPO
   - 全选：一键选中当前订单所有CPO
   - 智能合并：按交期自动合并同交期CPO
2. 唛架计划层数分配与递减
   - 唛架作为公共资源可被多个裁剪任务引用
   - 计划层数按分配自动递减
   - 强制校验防止超分配

**关键业务规则**：
- 规则1.1：唛架总计划层数 = 所有分配层数之和，超出禁止提交
- 规则1.2：层数递减 = 唛架总层数 - 已分配层数 = 剩余可用层数
- 规则1.3：新裁剪任务在"颜色"基础上新增"CPO"选择框
- 规则1.4：系统仅对所选CPO的订单数量创建裁剪任务

**数据流**：
```
生产订单 → 选择CPO(单/多/全选/智能合并) → 选择唛架 → 分配层数 → 
校验(层数≤剩余可用层数) → 创建裁剪任务
```

#### 模块2：床次任务与现场拉布模块

**核心功能**：

2.1 顶层任务（PC端）
- 创建裁剪任务作为"顶层拉布任务"，发布后PAD端可见
- 任务状态：未开始 → 拉布中 → 已完成
- 编辑控制：未开始时可编辑/删除；拉布中后隐藏编辑/删除按钮

2.2 现场床次创建（PAD端）
- 扫描/手动输入布卷号
- 输入拉布层数（正整数）
- 实时并发防超校验
- 提交完成后生成床次号

2.3 床次号规则
- 作用域：生产订单号内全局唯一
- 格式：床次-001、床次-002…（主布C开头，辅布F开头）
- 同秒提交以数据库记录ID为准保证严格递增

2.4 防超裁机制
- 预警（软提醒）：剩余层数<阈值（如10层）时弹提示
- 强控（禁止提交）：层数超计划时禁止创建/提交
- PC端超裁审核：放行（增加计划层数）或驳回（要求调整）

**状态机定义**：

裁剪任务状态流转：
```
未开始 ──(第一个床次开始拉布)──▶ 拉布中 ──(所有床次提交完成且剩余层数=0)──▶ 已完成
   │                               │
   └── 可编辑/删除                  └── 隐藏编辑/删除按钮
```

床次状态流转：
```
创建床次 ──(创建成功)──▶ 拉布中 ──(提交完成)──▶ 已拉布
                           │
                           └── 暂扣层数               └── 锁定层数
```

**并发防超计算公式**：
```
剩余可用层数 = 计划层数 - SUM(所有"已拉布"床次的锁定层数) - SUM(所有"拉布中"床次的暂扣层数)
```

#### 模块3：扎卡打印模块

**核心功能**：

| 场景 | 裁剪任务CPO状态 | 扎卡打印时是否显示CPO | 挂片管控策略 |
|------|----------------|---------------------|-------------|
| 场景A | 合并CPO（多个CPO） | 不显示CPO信息，打印通用扎卡 | 自由分配：系统根据各CPO交期顺序自动分配 |
| 场景B | 单一CPO | 强制显示CPO号，打印指定扎卡 | 强绑定：优先分配指定CPO，满额后按交期顺序自动继续 |

**分扎计件规则**：
- 合并CPO（无CPO信息）→ 按扎分配，只记录扎号和数量，不记录CPO
- 单一CPO（有CPO信息）→ 按CPO计件，每个CPO独立计数，扎卡印刷CPO号

#### 模块4：缝制挂片模块

**核心功能**：

4.1 自动充数模式（系统默认推荐）
- 合并CPO + 通用扎卡：系统按交期自动充数，满额后自动切换下一CPO
- 单一CPO + 指定扎卡：
  - 第一阶段：优先分配指定CPO
  - 第二阶段：指定CPO满额后，按交期顺序自动分配

4.2 手动充数模式
- 员工手动选择CPO挂片
- 系统展示未满额CPO列表（按交期排序）
- CPO满额后弹提示，要求切换
- 手动模式可跳过交期顺序，自由选择

4.3 模式切换
- 自动→手动：保存当前进度，提示切换成功
- 手动→自动：系统重新计算各CPO状态，按交期顺序自动选CPO

## 三、关键业务规则汇总

| 编号 | 规则描述 | 所属模块 |
|------|---------|---------|
| BR-001 | 唛架总计划层数 ≥ 所有裁剪任务已分配层数之和 | 裁剪任务 |
| BR-002 | 裁剪任务创建后，唛架剩余层数 = 总层数 - 已分配层数 | 裁剪任务 |
| BR-003 | 智能合并：同交期CPO自动归为一组 | 裁剪任务 |
| BR-004 | 床次号在生产订单号内全局唯一 | 床次任务 |
| BR-005 | 床次号提交完成时生成（非创建时） | 床次任务 |
| BR-006 | 剩余层数 = 计划层数 - 已锁定层数 - 暂扣层数 | 床次任务 |
| BR-007 | 输入层数 > 剩余可用层数 → 触发防超裁 | 床次任务 |
| BR-008 | 剩余层数 < 阈值 → 预警提示 | 床次任务 |
| BR-009 | 拉布中状态 → 隐藏编辑/删除按钮 | 床次任务 |
| BR-010 | 合并CPO → 扎卡不印CPO号，挂片自动分配 | 扎卡+挂片 |
| BR-011 | 单一CPO → 扎卡印CPO号，挂片优先分配该CPO | 扎卡+挂片 |
| BR-012 | 自动模式：按交期顺序自动选择CPO | 缝制挂片 |
| BR-013 | 手动模式CPO满额 → 禁止继续挂片，需切换CPO | 缝制挂片 |
| BR-014 | 手动模式可跳过交期顺序自由选择CPO | 缝制挂片 |

## 四、风险评估

| 风险项 | 风险等级 | 说明 |
|--------|---------|------|
| 并发超裁 | 高 | PAD端多小组同时创建床次，需严格防超 |
| 床次号唯一性 | 高 | 生产订单内全局唯一，并发场景需保证 |
| CPO合并后追溯 | 中 | 合并CPO的扎卡无CPO信息，追溯依赖挂片记录 |
| 模式切换数据一致性 | 中 | 自动/手动模式切换时需保证计数不丢失 |
| 超裁审核流程 | 中 | PC端审核增加操作步骤，需保证流程完整 |
| 交期变更影响 | 低 | CPO交期变更后已创建任务的处理 |

## 五、测试策略建议

### 5.1 测试重点
1. **唛架层数分配逻辑**：递减计算、超分配拦截
2. **并发床次创建**：多PAD同时创建床次的防超
3. **状态流转**：裁剪任务和床次的生命周期
4. **扎卡打印规则**：合并/单一CPO的分支逻辑
5. **挂片模式切换**：自动/手动模式的数据一致性
6. **超裁审核链路**：从PAD触发到PC审核完整流程

### 5.2 测试优先级
- **P0（阻塞级）**：层数防超、床次号唯一性、核心状态流转
- **P1（严重级）**：CPO合并逻辑、智能合并、挂片自动分配
- **P2（一般级）**：UI交互、提示信息、模式切换
- **P3（建议级）**：边界值、异常恢复、性能测试
