# 测试用例生成 — 验收标准

> 版本：1.0
> 使用方式：测试用例产出后逐项打勾，全部通过方可交付。

---

## 一、验收总则

验收分 **5 个维度、23 个检查项**，每个检查项有明确的 **PASS / FAIL** 判定条件。任一 FAIL 项必须修复后重新验收。

| 维度 | 检查项数 | 一票否决项 |
|------|----------|-----------|
| 1. 覆盖完整性 | 5 | #1.2 |
| 2. 准确性 | 5 | #2.1, #2.2 |
| 3. 可执行性 | 5 | #3.1 |
| 4. 规范性 | 5 | #4.1, #4.2 |
| 5. 可维护性 | 3 | 无 |

---

## 二、详细检查项

### 1. 覆盖完整性（5 项）

#### #1.1 需求全文可追溯

| 字段 | 内容 |
|------|------|
| 判定方式 | 打开「需求追溯表」，逐行核对需求文档每个章节 |
| PASS | 需求文档每个章节标题至少关联 1 个用例编号 |
| FAIL | 追溯表中缺少需求文档的某个章节 |

#### #1.2 通用覆盖矩阵无空白触发行 🔴 一票否决

| 字段 | 内容 |
|------|------|
| 判定方式 | 打开「通用覆盖矩阵自检」表，逐行检查 |
| PASS | A~I 每个维度，若「本模块是否触发」为「是」，则「已覆盖/待澄清」列非空（有检查点或 T-xx 占位） |
| FAIL | 任一触发行「已覆盖」列为空（静默跳过） |

**矩阵维度说明：**

| 维度 | 触发器（需求中出现即激活） |
|------|---------------------------|
| A 数据输入与校验 | 表单、请求体、查询参数、字段约束 |
| B 列表与查询 | 列表展示、搜索、筛选、分页、排序 |
| C 鉴权与角色 | 登录、管理员、角色区分 |
| D 资源标识 | 按 ID 访问详情、更新、删除 |
| E 时间与状态 | 活动时间、有效期、状态机、次数限制 |
| F 重复与一致性 | 提交/支付/下单、并发操作 |
| G 错误与可用性 | 校验失败提示、依赖不可用 |
| H 接口契约 | 提供 OpenAPI/Swagger 时叠加 |
| I 非功能附录 | 安全/性能要求（按需加载） |

#### #1.3 边界值全覆盖

| 字段 | 内容 |
|------|------|
| 判定方式 | 核对需求文档中每个数值阈值 |
| PASS | 每个阈值有对应的 V- 编号覆盖了 trio（阈值-1 / 阈值 / 阈值+1） |
| FAIL | 需求中「大于/小于/至少/最多/范围/限购 N 次」有任何一处缺 trio |

**边界值速查表：**

| 需求表述 | 必备用例 |
|----------|----------|
| X > N | N+1 (通过), N (边界), N-1 (另一侧) |
| X ≥ N | N-1 (拒绝), N (通过), N+1 (通过) |
| X < N | N-1 (通过), N (边界), N+1 (拒绝) |
| X ≤ N | N-1 (通过), N (通过), N+1 (拒绝) |
| min ~ max 范围 | min-1, min, min+1, max-1, max, max+1 |
| N 次限制 | N-1 (成功), N (成功), N+1 (拒绝) |
| 时间窗 start ~ end | start前/恰start/start后, end前/恰end/end后 |

#### #1.4 核心流程数量与结构一致

| 字段 | 内容 |
|------|------|
| 判定方式 | 对比需求预分析输出的流程数量与「核心流程测试」章节的流程数量 |
| PASS | 数量一致，且每条核心流程的树形缩进结构与预分析对齐；约 10～15 主节点 |
| FAIL | 数量不一致，或缩进结构被改乱 |

#### #1.5 待澄清已汇总闭环

| 字段 | 内容 |
|------|------|
| 判定方式 | 全文搜索 `T-` 编号，与文末「待澄清清单」逐条核对 |
| PASS | 每个 T-xx 在待澄清清单中有一条对应的问题，且写清「要问产品什么」 |
| FAIL | 存在 T-xx 在清单中缺失，或问题描述模糊（如仅写「待确认」） |

### 2. 准确性（5 项）

#### #2.1 每条测试点可判定 PASS/FAIL 🔴 一票否决

| 字段 | 内容 |
|------|------|
| 判定方式 | 全文扫描每条检查点，套用「三问自检」 |
| **三问** | ① 执行后能判断 PASS 还是 FAIL 吗？② 有明确的触发条件和可观察结果吗？③ 换一个测试执行人能否照着做？ |
| PASS | 三问全部回答「是」 |
| FAIL | 任一条检查点三问有一问回答「否」 |

#### #2.2 无功能描述冒充测试点 🔴 一票否决

| 字段 | 内容 |
|------|------|
| 判定方式 | 全文搜索以下禁止句式 |
| **禁止句式** | ①「系统支持…」②「需要通过…」③「旨在…」④「确保…」⑤「实现…功能」⑥「满足…需求」⑦「基于…架构」 |
| PASS | 全文无禁止句式 |
| FAIL | 任一处发现 → 必须改写为可验证的检查点 |

**改写对照：**

| ❌ 功能描述 | ✅ 可验证检查点 |
|-----------|---------------|
| 系统支持 JWT 登录 | 未带 Token 调用受保护接口 → 返回 401 |
| 需要通过 Redis 扣库存 | 库存为 1 时两次并发下单 → 仅 1 单成功，另 1 单返回库存不足 |
| 活动状态为进行中才可下单 | 活动 UPCOMING 时 POST 下单 → 返回约定错误码 |

#### #2.3 预期结果具体不模糊

| 字段 | 内容 |
|------|------|
| 判定方式 | 全文搜索模糊词 |
| **模糊词清单** | 「操作成功」「正常」「正确」「无误」「没问题」「输入合法数据」「合适的值」「合理的结果」 |
| PASS | 每条预期有具体值（状态码、字段名、订单状态、界面文案、错误码等） |
| FAIL | 发现模糊词 → 改写为具体断言 |

**改写对照：**

| ❌ 模糊 | ✅ 具体 |
|--------|--------|
| 输入合法数据，操作成功 | POST body name=`秒杀商品A` price=19.9，响应 201，商品状态为 `active` |
| 输入超长名称，返回错误 | name 为 51 个字符（上限 50），响应 400，错误信息指明 name 字段 |

#### #2.4 预期与需求业务规则一致

| 字段 | 内容 |
|------|------|
| 判定方式 | 抽查 20% 检查点的预期结果，与需求文档对应段落逐字对照 |
| PASS | 抽查项全部与需求文档一致 |
| FAIL | 任一项与需求文档冲突（需求写 A 但用例写 B） |
| 例外 | 需求文档自身矛盾时，有「（待澄清）」标注的不算 FAIL |

#### #2.5 无需求外的编造功能

| 字段 | 内容 |
|------|------|
| 判定方式 | 逐条检查点反向搜索需求文档，核对是否有对应功能描述 |
| PASS | 每条检查点的被测功能在需求文档中有依据（矩阵触发的 T-xx 占位也算有依据） |
| FAIL | 存在需求文档未写 + 矩阵未触发的检查点 |

### 3. 可执行性（5 项）

#### #3.1 用例评审表完整 🔴 一票否决

| 字段 | 内容 |
|------|------|
| 判定方式 | 检查「用例评审表」 |
| PASS | 表中包含所有检查点编号，每行有 5 列：编号、名称、优先级、类型、标签 |
| FAIL | 评审表缺失，或有编号遗漏，或列不完整 |

#### #3.2 编号唯一且格式正确

| 字段 | 内容 |
|------|------|
| 判定方式 | 从评审表中提取所有编号 |
| PASS | 每个编号格式为 `{前缀}-{两位序号}`（如 `V-01`、`F-02`），全表无重复 |
| FAIL | 编号格式错误（如 `V-1` 而非 `V-01`）或存在重复编号 |

**编号前缀规范：**

| 前缀 | 含义 | 前缀 | 含义 |
|------|------|------|------|
| F- | 核心流程 | U- | 更新/编辑 |
| L- | 列表/查询 | X- | 删除/作废 |
| D- | 详情/单条读 | V- | 校验/边界 |
| C- | 创建/新增 | P- | 权限/安全 |
| N- | 非功能（性能/安全） | T- | 待澄清占位 |

#### #3.3 优先级分布合理

| 字段 | 内容 |
|------|------|
| 判定方式 | 统计评审表中各优先级数量 |
| PASS | P0 ≤ 20%，P1 ≈ 40%，P2 ≈ 30%，P3 ≤ 10%（允许 ±5% 浮动） |
| FAIL | P0 占比 > 25%，或出现 P0 用于次要场景 |

**优先级定义：**

| 级别 | 定义 | 建议占比 |
|------|------|----------|
| P0 | 核心流程、资金/权限、数据丢失风险 | ≤ 20% |
| P1 | 主功能正向与关键异常 | ≈ 40% |
| P2 | 边界、次要场景 | ≈ 30% |
| P3 | 低概率、体验类 | ≤ 10% |

#### #3.4 前置条件可复现

| 字段 | 内容 |
|------|------|
| 判定方式 | 扫描需要特殊数据/环境的检查点 |
| PASS | 每条特殊前提写明：所需账号角色、数据准备方式、环境条件 |
| FAIL | 存在「特殊环境」「特定数据」等笼统表述，未写清具体需求 |

#### #3.5 接口类检查点可直接转化为自动化

| 字段 | 内容 |
|------|------|
| 判定方式 | 对接口类检查点，检查是否包含 HTTP 方法、路径、关键请求体/参数 |
| PASS | API 检查点写明了 method + path + 关键字段，可直接作为自动化脚本输入 |
| FAIL | 接口检查点仅有业务描述，无具体 API 信息 |

### 4. 规范性（5 项）

#### #4.1 文档章节结构完整 🔴 一票否决

| 字段 | 内容 |
|------|------|
| PASS | 以下 8 个章节全部存在（无核心流程可省略第④项但须注明）： |
| | ① 文档信息 ② 知识库检索摘要 ③ 需求追溯表 ④ 核心流程测试 |
| | ⑤ 详细功能点测试 ⑥ 用例评审表 ⑦ 通用覆盖矩阵自检 ⑧ 待澄清清单 |
| FAIL | 任一必填章节缺失 |

#### #4.2 Markdown 层级结构正确 🔴 一票否决

| 字段 | 内容 |
|------|------|
| 判定方式 | 检查标题层级是否与文档信息中声明的「结构说明」一致 |
| **简单型**（< 10 检查点） | H2（功能）→ H3（检查点） |
| **标准型**（10～30 检查点） | H2（功能）→ H3（场景）→ H4（检查点） |
| **复杂型**（> 30 检查点） | H2（功能）→ H3（子模块）→ H4（场景）→ H5（检查点） |
| PASS | 标题层级 ≤ 4 级；无单检查点的空壳嵌套；与声明的结构类型一致 |
| FAIL | 出现 5 级及以上标题；声明标准型却用二级结构；有「测试用例→用例1→步骤1」无业务语义的分组 |

#### #4.3 术语全文统一

| 字段 | 内容 |
|------|------|
| 判定方式 | 全文搜索术语使用一致性 |
| **统一术语** | 缺陷（非 Bug）、检查点/测试点（非 Case）、PASS/FAIL（非 pass/fail）、待澄清（非 待确认/待定）、预期结果（非 expected_result） |
| PASS | 同一概念全文使用相同术语 |
| FAIL | 同一概念出现多种不同表述 |

#### #4.4 检查点标题命名规范

| 字段 | 内容 |
|------|------|
| 判定方式 | 抽取所有 `####` 和 `#####` 标题 |
| PASS | 标题为动宾或主谓结构，可脱离上下文独立理解（如「价格为负数时拒绝保存」）；非「测试1」「用例2」 |
| FAIL | 存在无意义编号标题，或标题与内容不符 |
| **正例** | `#### 未登录访问商品详情` `#### 错误密码登录返回 401` |
| **反例** | `#### 测试1` `#### 用户输入合法用户名并且系统验证通过` |

#### #4.5 检查点书写格式正确

| 字段 | 内容 |
|------|------|
| 判定方式 | 检查每条检查点的书写风格 |
| PASS | 用自然中文 bullet 描述，融合条件 + 操作 + 预期；不使用「前置条件：」「操作：」「预期：」标签分割 |
| FAIL | 大量使用标签分割的机械模板格式 |
| **正例** | `- admin 已登录，POST /api/products body: name="商品A" price=19.9，响应 201，商品状态为 active` |
| **反例** | `- 前置条件：admin 已登录。操作：POST 创建商品。预期：返回成功。` |

---

### 5. 可维护性（3 项）

#### #5.1 标签体系一致

| 字段 | 内容 |
|------|------|
| 判定方式 | 统计评审表中所有标签，检查命名约定是否统一 |
| PASS | 标签采用「模块主标签 + 子标签 + 场景标签」模式（如 `products,list,status`），同一概念标签一致 |
| FAIL | 同一概念用了不同标签（如 `create` 和 `creation` 混用），或颗粒度混乱 |

**类型取值：** `smoke_sanity` | `functional` | `regression` | `boundary` | `security` | `integration` | `performance` | `pending_clarification`

#### #5.2 知识库来源已标注

| 字段 | 内容 |
|------|------|
| 判定方式 | 检查所有从知识库/历史经验补充的检查点 |
| PASS | 每条知识库补充的检查点在追溯表或正文中标明来源类型（如「历史用例·鉴权设计」「历史缺陷·#892」） |
| FAIL | 知识库摘要有命中记录，但正文无任何引用标注 |

#### #5.3 可追溯到需求段落

| 字段 | 内容 |
|------|------|
| 判定方式 | 随机抽取 5 条检查点，反向在需求文档中定位对应描述 |
| PASS | 5 条全部能在 30 秒内定位到需求文档对应段落 |
| FAIL | 任一条无法在需求文档中找到来源 |

## 三、问题分级与处理

| 级别 | 定义 | 对应检查项 | 处理 |
|------|------|----------|------|
| 🔴 阻塞级 | 缺此不视为完成 | #1.2 #2.1 #2.2 #3.1 #4.1 #4.2 | **必须修改，否则不予交付** |
| 🟡 严重级 | 影响质量但可降级交付 | #1.1 #1.3 #1.4 #2.3 #2.4 #2.5 #3.2 #3.3 | **建议修改，发布前复核** |
| 🟢 一般级 | 规范/体验问题 | #1.5 #3.4 #3.5 #4.3 #4.4 #4.5 #5.1 #5.2 #5.3 | **可择期修改，不阻塞当前迭代** |

## 四、验收流程

```
用例产出
  ↓
【自检】生成者逐项核对（6 类自检 + 修复）
  ↓
【提交评审】交付物 + 自检结果一并提交
  ↓
【评审人验收】逐项核对本文 23 个检查项
  ↓
  ├── 全部 PASS → ✅ 验收通过
  ├── 仅 🟢 未通过 → ⚠️ 有条件通过，记录问题清单
  └── 🔴 或 🟡 未通过 → ❌ 退回修改，附带问题清单
        ↓
     修改后重新提交验收（最多 3 轮）
```

## 六、快速验收卡（7 问，日常轻量用）

| # | 问 | PASS 条件 |
|---|-----|----------|
| 1 | 覆盖矩阵触发行全填了吗？ | 无空白行 |
| 2 | 每个阈值有 trio 吗？ | 全部有 V-xx 覆盖 |
| 3 | 有「系统支持…」句式吗？ | 全文无 |
| 4 | P0 超 20% 了吗？ | ≤ 20% |
| 5 | 知识库章节还在吗？ | 存在且非空（含「无命中」说明） |
| 6 | 待澄清汇总了吗？ | T-xx 全在文末清单中 |
| 7 | 8 个章节缺不缺？ | 全部存在 |

> 7 问全 PASS → 快速放行。任一 FAIL → 退回走完整 23 项验收。

## 七、交付文档标准模板

测试用例文档必须包含以下结构：

```markdown
# [模块名] 测试用例

## 文档信息
（依据需求、生成版本、知识库检索状态、测试范围、结构说明、待澄清汇总数）

## 知识库检索摘要
（检索结果：命中 N 条的具体内容，或「无命中」）

## 需求追溯表
| 需求段落 | 用例编号范围 | 覆盖说明 |

## 核心流程测试
### 流程1：[端到端业务目标]  <!-- F-01 -->
- [主步骤] → [子步骤] → [断言]

## 详细功能点测试
## [功能模块]
### [场景分组]
#### [检查点标题]  <!-- V-01 P0 -->
- [条件 + 操作 + 预期，bullet 自然融合]

## 用例评审表
| 编号 | 名称 | 优先级 | 类型 | 标签 |

## 通用覆盖矩阵自检
| 维度 | 触发? | 已覆盖/待澄清 | 备注 |
（A~I 逐维度打勾）

## 边界值速查表
| 需求原文 | 阈值类型 | 阈值 | 必测值 | 对应编号 |

## 待澄清清单
| # | 问题 | 关联编号 | 优先级 |
```
