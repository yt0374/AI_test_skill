# 接口自动化测试用例 — 验收标准

> 版本：1.0
> 使用方式：接口自动化脚本产出后逐项打勾，全部通过方可交付。

---

## 一、验收总则

验收分 **6 个维度、27 个检查项**，每个检查项有明确的 **PASS / FAIL** 判定条件。

| 维度 | 检查项数 | 一票否决项 |
|------|----------|-----------|
| 1. Spec 规格正确性 | 6 | #1.1, #1.2 |
| 2. 用例覆盖完整性 | 6 | #2.1, #2.3 |
| 3. 链式执行与变量池 | 5 | #3.1, #3.4 |
| 4. 断言与验证 | 4 | #4.1, #4.2 |

## 二、详细检查项

---

### 1. Spec 规格正确性（6 项）

#### #1.1 YAML frontmatter 语法合法 🔴 一票否决

| 字段 | 内容 |
|------|------|
| 判定方式 | 提取 spec 文件 `---` 之间的内容，用 YAML 解析器验证 |
| PASS | YAML 解析成功，无语法错误 |
| FAIL | 解析抛出异常（缩进不对、引号不匹配、非法字符） |
| 验证命令 | `python -c "import yaml; yaml.safe_load(open('spec.md').read().split('---')[1])"` |

#### #1.2 必填字段齐全且合法 🔴 一票否决

| 字段 | 必填 | 合法取值/约束 |
|------|:---:|------|
| `kind` | ✅ | `api_pytest`（固定值） |
| `base_url` | ✅ | 非空 HTTP/HTTPS URL，不能是生产地址 |
| `cases` 或 `scenarios` | ✅ | 至少存在一个且非空数组 |
| `setup` | — | `test_init`（默认）/ `none` / `{kind: data_factory, script: ...}` |
| `auth` | — | `{type: bearer_login, login_path, username_field, password_field, username_env, password_env}` |
| `timeout_default` | — | 正整数，默认 30(秒) |

| PASS | 所有必填字段存在且取值合法；`base_url` 非生产地址 |
| FAIL | 任一必填字段缺失；`base_url` 为空或指向生产；`kind` 不为 `api_pytest` |

#### #1.3 每个 case 的必填子字段齐全

| case 子字段 | 必填 | 说明 |
|-------------|:---:|------|
| `id` | ✅ | 唯一标识，建议格式 `TC-<模块>-<NNN>` |
| `method` | ✅ | `GET` / `POST` / `PUT` / `PATCH` / `DELETE` |
| `path` | ✅ | 以 `/` 开头的接口路径 |
| `expect_status` | ✅ | 整数，预期 HTTP 状态码（200/201/400/401/403/404/422/500 等） |

| PASS | 每个 case 的 `id` + `method` + `path` + `expect_status` 四字段齐全，且 `id` 全表唯一 |
| FAIL | 任一 case 缺上述字段，或出现重复 `id` |

#### #1.4 scenarios 结构正确

> 当使用 `scenarios[]`（多步场景）代替或补充 `cases[]` 时检查此项。

| 字段 | 必填 | 说明 |
|------|:---:|------|
| `name` | ✅ | 场景名，须可读 |
| `steps[]` | ✅ | 至少 2 个 step |
| step.`method` | ✅ | HTTP 方法 |
| step.`path` | ✅ | 接口路径，支持 `{变量名}` 引用上游捕获值 |
| step.`expect_status` | ✅ | 预期状态码 |
| step.`capture` | — | `{变量名: JSON路径}`，用于步骤间传值 |

| PASS | 每个 scenario 有 name + steps，steps ≥ 2，每个 step 四字段齐全 |
| FAIL | scenario 无 name；steps 为空或仅 1 步；step 缺必填字段 |

#### #1.5 占位符使用正确

| 占位符 | 含义 | 约束 |
|--------|------|------|
| `{{timestamp}}` | 秒级时间戳 | 仅用于 `json_body` 中需唯一化的字段 |
| `{{uuid}}` | UUID4 前 8 位 | 同上 |
| `${变量名}` | 变量池引用 | 变量名必须在 `extract_vars` / `capture` 或 auth 登录中有产出 |
| `__ENV__:VAR` | 环境变量 | 对应的环境变量必须在 `.env` 或 CI 中定义 |

| PASS | 所有 `${变量名}` 的变量有明确来源；占位符不用于 `id` 字段（id 必须稳定） |
| FAIL | `${变量名}` 引用了不存在的变量（无 extract_vars/capture 产出该变量） |

#### #1.6 数据驱动配置正确

> 仅当 spec 使用了 `data_source`（CSV/JSON 参数化）时检查此项。

| PASS | `data_source` 指向的文件存在；CSV 表头与 `json_body` 中 `${xxx}` 占位符一致；CSV 中 `${expect_status}` 列的值均为合法 HTTP 状态码 |
| FAIL | data_source 文件不存在；CSV 表头与 json_body 占位符不匹配；`expect_status` 列含非数字值 |
| 验证命令 | `head -3 <data_source文件>` 核对表头 |

### 2. 用例覆盖完整性（6 项）

#### #2.1 底线：每个端点 H + X 双覆盖 🔴 一票否决

接口测试用例按风险分层覆盖，代号含义：

| 代号 | 含义 | 最低要求 |
|------|------|----------|
| **H** (Happy Path) | 主路径成功 | **每个端点至少 1 条**成功用例 |
| **X** (eXception) | 协议/业务拒绝 | **每个端点至少 1 条**失败用例（401/403/400/422/404 等） |
| **E** (Edge / Contract) | 契约边界 | 有接口文档时：必填缺失、类型错误、枚举非法、超长字段 |
| **S** (Security) | 低成本安全 | 匿名访问、过期 Token、水平越权 |
| **P** (Performance sanity) | 粗看耗时 | 单次响应耗时量级检查 |

| PASS | 每个被测端点都有 ≥1 条 H（成功）且 ≥1 条 X（失败） |
| FAIL | 任一端点只有成功用例无失败用例，或反之 |

#### #2.2 S（安全）覆盖

> 被测接口涉及资金/订单/用户数据时必检；普通查询接口为加分项。

| 安全场景 | 必测项 |
|----------|--------|
| 匿名访问受保护接口 | `no_auth: true` + `expect_status: 401` |
| 普通用户访问管理员接口 | 普通用户 token + `expect_status: 403` |
| 过期/篡改 Token | 伪造 Token + `expect_status: 401` |
| 水平越权 | 用户 A 访问用户 B 的资源 → 403 或 404 |

| PASS | 资金/订单/用户相关接口至少覆盖上述 3 项；普通接口至少有「匿名访问受保护接口」 |
| FAIL | 资金/订单类接口的安全用例为 0 |

#### #2.3 E（契约边界）覆盖 🔴 一票否决

> 当接口文档（OpenAPI/Swagger/手写）定义了字段约束时必检。

| 边界场景 | 必测项 |
|----------|--------|
| 必填字段缺失 | 每个必填字段各 1 条，期望 400/422 |
| 类型错误 | 数值字段传字符串 → 400/422 |
| 枚举非法值 | 枚举字段传未定义值 → 400/422 |
| 超长字符串 | 字段超过 maxLength → 400/422 |

| PASS | 有接口文档时：每个端点至少覆盖必填缺失 + 类型错误；无接口文档时：从需求文档提取字段约束并逐条覆盖 |
| FAIL | 有接口文档但边界用例为 0 |

#### #2.4 依赖关系正确建模

| PASS | ① 认证链正确：登录接口先于受保护接口执行；② 路径参数引用正确：POST 创建返回 ID → GET 使用该 ID 查询；③ 无循环依赖（A→B→C→A） |
| FAIL | 出现「先调受保护接口再登录」的反向依赖；`depends_on` 引用了不存在的 case_id；存在 A→B→A 的循环依赖 |

**4 种接口依赖关系识别：**

| 依赖类型 | 判定规则 | 置信度 |
|----------|----------|--------|
| 认证链 | 下游接口需要 auth，上游返回 token | 高 |
| 路径参数引用 | 下游路径参数 = 上游响应字段 | 高 |
| 请求体引用 | 下游请求体字段 = 上游响应字段 | 中 |
| 命名启发式 | 路径前缀相同 + 资源名匹配 | 低 |

#### #2.5 测试数据唯一且可重复执行

| PASS | 创建类用例的 `json_body` 中唯一字段（username / name / email）使用了 `{{timestamp}}` 或 `{{uuid}}` 确保每次运行不冲突；或 spec 注明需配合 `setup: test_init` 重置数据 |
| FAIL | 创建类用例使用硬编码唯一字段（如 `username: "testuser"`），第二次运行必然因重复而失败 |

#### #2.6 接口覆盖率达标

| 覆盖率指标 | 阈值 |
|------------|------|
| 端点覆盖率 | ≥ 80%（被测端点数 / 接口文档定义端点数） |
| HTTP 方法覆盖率 | GET + POST 必须覆盖；PUT/PATCH/DELETE 若文档中有则至少 1 条 |

| PASS | 端点覆盖率 ≥ 80%；所有 HTTP 方法有对应用例 |
| FAIL | 端点覆盖率 < 80% 且无合理解释；存在 DELETE 端点但无测试 |

### 3. 链式执行与变量池（5 项）

#### #3.1 extract_vars / capture 与变量引用闭环 🔴 一票否决

| PASS | spec 中每个 `${变量名}` 都能追溯到以下来源之一： |
| | ① 某个 case 的 `extract_vars` 产出 |
| | ② 某个 scenario 步骤的 `capture` 产出 |
| | ③ auth 登录自动写入的 `token` |
| | ④ 系统内置变量（`timestamp` / `uuid`） |
| FAIL | 存在 `${变量名}` 无任何来源（孤引用） |
| 验证方式 | grep 所有 `\${...}` 与所有 `extract_vars`/`capture` 输出做差集 |

#### #3.2 depends_on 链完整且无断路

| PASS | ① 每个 `depends_on: [TC-xxx]` 引用的 case_id 在 spec 中存在；② 不存在循环依赖；③ 失败跳过逻辑正确：上游 case 失败时下游自动 skip，不产生无效失败 |
| FAIL | 引用了不存在的 case_id；存在循环依赖 |

**变量池运行机制：**

```
① POST /login → 200, extract_vars: { token: $.data.token }
② POST /orders → depends_on: [①], headers: { Authorization: Bearer ${token} }
③ GET /orders/{order_id} → depends_on: [②], path: /orders/${order_id}

若 ① 失败 → ②③ 自动 skip，报告中只记录 ① 为失败，不误报
```

#### #3.3 auth 配置与 no_auth 不矛盾

| PASS | ① 若 spec 声明了 `auth: bearer_login`：需鉴权的 case 不写 `no_auth: true`；不需鉴权的 case（如注册、登录、匿名接口）明确写 `no_auth: true`；② `login_path` 指向的端点确实返回 token（对应 `token_json_path` 字段） |
| FAIL | 对同一个 case 既配置了 auth 又写了 `no_auth: true`（矛盾）；需要 token 的 case 未写 auth 且上游也没有 extract token |

#### #3.4 无硬编码敏感信息 🔴 一票否决

| PASS | spec 中无硬编码的真实密码、生产 Token、生产密钥；敏感值通过 `__ENV__:VAR` 或 `username_env`/`password_env` 从环境变量注入 |
| FAIL | spec frontmatter 或 json_body 中出现明文真实密码或生产 Token |

#### #3.5 生成脚本可独立运行

| PASS | 生成命令执行成功，且 `--collect-only` 返回所有 test_ 函数，无 import 错误 |
| FAIL | 生成失败或 `--collect-only` 有导入错误 |
| 验证命令 | `./generate.sh <spec> && cd output && python -m pytest <spec>.py --collect-only -q` |

### 4. 断言与验证（4 项）

#### #4.1 每条 case 有 expect_status 🔴 一票否决

| PASS | spec 中每个 case 和 scenario 的每个 step 都有明确的 `expect_status` 字段 |
| FAIL | 任一 case 或 step 缺少 `expect_status` |

#### #4.2 正向用例断言不止于状态码 🔴 一票否决

| PASS | 正向（expect_status: 200/201）用例至少包含一种额外断言： |
| | ① `expect_json_subset` — 验证 JSON 响应中关键字段值 |
| | ② `expect_body_substrings` — 验证响应体包含特定字符串 |
| | ③ `extract_vars` — 提取字段供下游使用（变相验证字段存在且路径正确） |
| FAIL | 正向用例仅有 `expect_status: 200`，无任何业务字段断言 |

**断言类型速查：**

| 断言类型 | spec 字段 | 说明 |
|----------|----------|------|
| 状态码 | `expect_status: 200` | HTTP 状态码比较 |
| JSON 字段值 | `expect_json_subset: {field: value}` | 响应 JSON 中指定字段的值比较 |
| 响应体子串 | `expect_body_substrings: ["子串"]` | 验证响应体包含特定文本 |
| 响应时间 | `timeout` 和 `expect_response_time_ms` | 耗时阈值检查 |
| 响应头 | `expect_headers: {key: value}` | 响应头字段校验 |

#### #4.3 断言值与接口文档一致

| PASS | ① 成功状态码与接口文档 responses 定义一致（200/201/204）；② 失败状态码与业务规则一致（401/403/404/400/422）；③ `expect_json_subset` 中的字段名与接口文档 schema 一致 |
| FAIL | 状态码与接口文档冲突（如文档写 201 但 spec 写 200）；字段名拼写错误 |

#### #4.4 超时配置合理

| PASS | `timeout_default` 为合理值（默认 30s，一般 10~120s）；若有显式 `expect_response_time_ms`，其值 ≤ `timeout_default * 1000` |
| FAIL | `timeout_default` < 1000ms（不可能完成）；`expect_response_time_ms` > `timeout_default`（永远超时） |

### 5. 自愈与鲁棒性（3 项）

#### #5.1 自愈策略与故障类型匹配

接口失败应先分类再处理——能自愈的不报缺陷。

| 故障类型 | 策略 | 最大重试 | 延迟方式 |
|----------|------|----------|----------|
| 401 Unauthorized | 刷新 Token | 5 次 | 固定 0.5s |
| 500 Server Error | 指数退避 + 抖动 | 3 次 | 1s × 2^attempt + 随机抖动 |
| 429 Rate Limit | 指数退避 + 抖动 | 3 次 | 2s × 2^attempt + 随机抖动 |
| 请求超时 | 调整超时时间 | 2 次 | 每次 +10s，最大 120s |
| 连接失败 | 指数退避 + 抖动 | 3 次 | 2s × 2^attempt + 随机抖动 |

| PASS | 执行日志中自愈重试次数 ≤ 上表最大值；400 参数错误不自愈（直接失败）；自愈成功的用例标记 healed 且不上报缺陷 |
| FAIL | 重试次数超出上限；对 400 参数错误执行了重试；自愈成功仍被报为缺陷 |

**随机抖动说明**：多个请求同时遇到 500，若同一时刻全部重试会再次打挂服务（惊群效应）。延迟 × (0.5~1.5 的随机系数) 可错开重试时间。

#### #5.2 失败根因分析产出完整

| PASS | 若有失败用例，分析报告包含：① 每条失败归类（环境/数据/代码/测试设计/未知）；② 置信度（0~1）；③ 去重指纹（接口路径 + 状态码 + 错误信息前 50 字符的哈希）；④ 修复建议不为空 |
| FAIL | 分析报告缺失关键字段；所有失败全归为「未知」且无人工标注 |

**5 类失败归因：**

| 类别 | 说明 | 典型特征 |
|------|------|----------|
| 环境 | 网络抖动、服务重启、Token 过期 | Connection refused、502/504、证书错误 |
| 数据 | 数据缺失、格式错误、唯一约束冲突 | 404 资源不存在、409 冲突 |
| 代码 | 服务端 Bug、逻辑错误 | 500、空指针、业务逻辑异常 |
| 测试设计 | 用例写错、断言不对、数据构造有误 | AssertionError、400 参数格式错误 |
| 未知 | 无法自动判断 | 需人工排查的异常 |

#### #5.3 去重指纹生效

| PASS | 同一接口 + 同一状态码 + 同一错误信息 → 只生成 1 条缺陷记录，后续标记为重复跳过 |
| FAIL | 相同指纹的失败重复生成多条缺陷记录 |

**去重算法**：`指纹 = MD5(接口路径 + HTTP状态码 + 错误信息前50字符)`

### 6. 交付与 CI 就绪（3 项）

#### #6.1 产物完整可追溯

| 必含产物 | 说明 |
|----------|------|
| Spec 源文件 | `.md` 含 YAML frontmatter |
| 生成的 pytest 脚本 | 可独立 `pytest` 执行 |
| 环境变量说明 | 列出所有需要的环境变量及其含义 |
| 执行报告 | 至少一种机器可读格式（JUnit XML / JSON / Allure） |

| PASS | 以上 4 件齐全 |
| FAIL | 缺 spec 源文件（只有生成的脚本不可追溯）；缺环境变量说明导致他人无法复现 |

#### #6.2 CI 命令行可一键执行

| PASS | 存在一条可放入 CI 的单行命令，且执行后输出明确的通过率（passed / failed / skipped 数量）；不需要手动改代码中的 `BASE_URL` |
| FAIL | 需要手动修改脚本中的硬编码地址才能跑；需要手动创建数据库/依赖服务且这些步骤未文档化 |

#### #6.3 失败自动分流

| PASS | 执行后红色用例自动产生：① 根因分析报告（归类 + 置信度 + 修复建议）；② 缺陷草稿（含复现步骤和请求响应）；③ 去重后的缺陷清单（同一问题不重复记录） |
| FAIL | 有失败用例但无任何分析输出；相同问题重复报了多个缺陷 |

## 三、问题分级与处理

| 级别 | 定义 | 对应检查项 | 处理方式 |
|------|------|----------|----------|
| 🔴 阻塞级 | 缺此不视为完成 | #1.1 #1.2 #2.1 #2.3 #3.1 #3.4 #4.1 #4.2 | **必须修改，否则不予交付** |
| 🟡 严重级 | 影响质量但可降级交付 | #1.3 #1.4 #1.5 #1.6 #2.2 #2.4 #2.5 #3.2 #3.3 #4.3 #5.1 | **建议修改，发布前复核** |
| 🟢 一般级 | 规范/优化问题 | #2.6 #3.5 #4.4 #5.2 #5.3 #6.1 #6.2 #6.3 | **可择期修改，不阻塞当前迭代** |

## 四、验收流程

```
Spec 编写完成
  ↓
【自检】YAML 语法 + 字段完整性 + 占位符闭环
  ↓
【生成验证】执行代码生成，确认产出 .py 脚本
  ↓
【静态验证】--collect-only 确认所有用例可被 pytest 发现
  ↓
【执行验证】目标环境或本地跑一遍，收集通过率和失败分析
  ↓
【提交评审】spec 源文件 + 生成脚本 + 执行报告一并提交
  ↓
【评审人验收】逐项核对本文 27 个检查项
  ↓
  ├── 全部 PASS → ✅ 验收通过
  ├── 仅 🟢 未通过 → ⚠️ 有条件通过，记录问题清单
```

## 七、api_pytest Spec 字段速查

```yaml
---
kind: api_pytest                          # 必填，固定值
base_url: http://127.0.0.1:5000           # 必填，不能是生产地址
setup: test_init | none | {kind, script}  # 可选，默认 test_init
auth:                                     # 可选，Bearer 登录
  type: bearer_login
  login_path: /api/auth/login
  username_field: username
  password_field: password
  username_env: TEST_USERNAME
  password_env: TEST_PASSWORD
  token_json_path: token                  # 可选，token 在响应 JSON 中的路径
timeout_default: 30                       # 可选，默认 30s

cases:                                    # cases[] 与 scenarios[] 至少其一
  - id: TC-XXX-001                        # 必填，全表唯一
    name: "用例名称"                       # 可选
    method: GET|POST|PUT|PATCH|DELETE     # 必填
    path: /api/xxx                        # 必填，以 / 开头
    query: {key: value}                   # 可选，URL 查询参数
    json_body: {field: value}             # 可选，请求体 JSON
    headers: {Authorization: "Bearer ${token}"}  # 可选
    no_auth: true                         # 可选，跳过认证
    expect_status: 200                    # 必填
    timeout: 30                           # 可选，单 case 超时
    extract_vars:                         # 可选，提取响应字段入变量池
      access_token: $.data.token          # 格式：{变量名: JSONPath}
    depends_on: [TC-XXX-001]              # 可选，依赖上游 case
    data_source: data/xxx.csv             # 可选，CSV 数据驱动
    expect_body_substrings: ["子串"]       # 可选，响应体包含检查
    expect_json_subset: {field: value}    # 可选，JSON 字段值检查

scenarios:                                # 多步场景（与 cases 可并存）
  - name: "场景名"                         # 必填
    steps:
      - method: POST
        path: /api/xxx
        json_body: {...}
        expect_status: 201
        capture:                          # 步间传值
          var_name: json.path
      - method: GET
        path: /api/xxx/{var_name}
        headers: {Authorization: "Bearer {var_name}"}
        expect_status: 200
---
```
