# ERP Requirements Analysis & Test Case Generator

Analyze business requirements for the **瑞晟智能制造协同平台** and generate complete test
deliverables: requirements specification → test coverage matrix → data-driven test cases →
automation scripts.

## Core Principle: Data-Driven Testing

ERP systems are fundamentally data-intensive — the same business logic applies to many
different entities (customers, materials, suppliers, warehouses). A single requirement like
"validate credit limit" must work for hundreds of customers with different credit profiles.
Testing with just one data point is insufficient.

This skill therefore treats **test data as a first-class deliverable**, not an afterthought.
Every phase extracts, catalogs, and leverages real data from the ERP system to ensure tests
cover the full range of business scenarios.

## System Context

The ERP serves the garment/apparel manufacturing industry with 14 sidebar modules.
Core business data is captured from 9 modules in `module_texts.txt` (project root or
`Documents/ERP-test/`). **Always read this file first.** It is the single source of truth
for:
- **Page structure**: Menu paths, page names, tab labels
- **Entity instances**: Real customer names (gm.销售#1), material codes (ML863, ML870),

## Phase 1: Requirements Analysis + Data Discovery

### Step 1.1 — Understand the Input

Read any files the user provides. If the user describes requirements conversationally,
treat the conversation as the source material. Also read `module_texts.txt` to ground
the analysis in the actual ERP module structure.

### Step 1.2 — Extract Requirements

From the input, extract and organize:

- **Functional Points (功能点)**: What the system must do. Each point is a discrete
  capability: "user can create a sales order", "system validates inventory before
  confirming order".
- **Business Rules (业务规则)**: Constraints and logic. "A sales order must have a
  valid customer ID", "采购订单金额超过10万需要主管审批".
- **Data Flow (数据流转)**: How data moves between modules. "Sales order → Production
  order → Material requirement → Purchase order → Receiving → Warehouse".
- **Actors/Roles (角色)**: Who interacts with the system. Operators, managers, external
  suppliers, warehouse staff.
- **Preconditions & Postconditions (前置/后置条件)**: What must be true before and
  after each function.
- **Edge Cases & Exceptions (边界/异常)**: Error conditions, boundary values, special
  handling.

### Step 1.3 — Map to ERP Modules & Extract State Machines

Match each functional point to the specific ERP module(s) it belongs to. Use the
module structure from `module_texts.txt`. A single requirement may span multiple
modules — flag these as cross-module concerns.

**State Machine Analysis:** For every entity involved in the requirements, identify
its lifecycle states and valid transitions. State machines drive test scenarios:
every transition is a test case, every invalid transition is a boundary test.

Use `references/erp_domain_knowledge.md` §1 for the catalog of documented state
machines. When analyzing new requirements, compare against known states to identify
gaps or new states being introduced.

Example — if the requirement involves a purchase order, the state machine is:
```
草稿 --[生效]--> 生效 --[确认到货]--> locks receipt notices
草稿 --[删除]--> only in 草稿
```
This yields at minimum: TC for 生效, TC for 确认到货, TC for 删除草稿, TC for
attempting to delete 生效 (should fail).

### Step 1.5 — Extract Test Data Sources

This is a critical step. `module_texts.txt` contains real ERP entity data embedded
in the captured page text. Extract and catalog this data systematically.

**What to extract:**

Scan `module_texts.txt` for these entity types and build a catalog:

| Entity Type | How to Identify | Examples from module_texts.txt |
|-------------|----------------|-------------------------------|
| **Users/Roles** | `gm.模块名#N` patterns | gm.销售#1, gm.采购#2, gm.仓库#1, gm.生产#3 |
| **Materials/Style Numbers** | `ML` prefix codes, style IDs | ML863, ML870 |
| **Customers** | Names in sidebar/user lists | ZOU WENQIANG, CHEN FEIMAN |
| **Suppliers** | Names in 数据 module | (extracted from 供应商 section) |
| **Warehouse Types** | 面料/松布/辅料/裁片/成品/半成品 + 仓库 | 面料仓库, 辅料仓库, 成品仓库 |
| **Status Values** | Workflow states visible in page text | Draft, Issued, Counting, Completed |
| **Date/Time Formats** | Timestamp patterns | 2026/06/30 08:00:13 |
| **ID Conventions** | Numeric ID patterns | 108485, 100214 |

**How to catalog — output `test_data_inventory.md`:**

```markdown
| Code | Description | Related Modules | Source |
|------|-------------|----------------|--------|
| ML863 | 生产订单样衣 | 生产, 销售 | module_texts.txt §01 |

### Data Variations for Testing
For each entity type, identify the distinct variation categories relevant
to the requirements being analyzed. Examples:

**Customer credit profiles** (for credit check testing):
| Category | Description | Example Customers | Expected Behavior |
|----------|-------------|-------------------|-------------------|
| Normal credit | Has sufficient credit | 品牌A, 品牌B | Order passes |
| Zero credit | Credit limit = 0 | 品牌C | Order blocked |
| No credit record | New customer, no credit data | 品牌D | Use default/ask |

**Warehouse stock levels** (for inventory check testing):
| Category | Description | Example | Expected Behavior |
|----------|-------------|---------|-------------------|
| Sufficient | Stock > order qty | ML863 @ 面料仓: 500m | Pass |
| Boundary | Stock at 80% threshold | ML870 @ 辅料仓: 100 units | Pass (exact boundary) |
| Insufficient | Stock < order qty | ML863 @ 裁片仓: 5 units | Block |
```

**Why this matters:** ERP testing without real data is shallow. The data inventory:
1. Gives testers concrete values instead of placeholders like `<test_customer>`
2. Reveals data variation categories that the requirements didn't explicitly mention
3. Feeds directly into parametrized test cases in Phase 3 and 4

## Phase 2: Test Coverage Matrix (with Data Dimensions)

### Step 2.1 — Map Requirements to Test Scenarios

For each functional point (F001, F002...), define what test scenarios are needed to
verify it. Each scenario gets a unique ID and links back to the requirement it covers.

**State-Machine-Driven Scenarios:** For every entity with states, generate one test
per valid transition (P1) and one test per invalid transition attempt (P2). Example:
if the requirement involves WMS material receipt, the state machine from
`references/erp_domain_knowledge.md` §1.6 dictates scenarios for each status change.

**Formula-Driven Boundary Tests:** For every business formula, generate tests at
exact threshold values, threshold±1, zero, and extreme values. Example: the box
type allocation rule (qty ≤ 1.2上限 → 1.2货位, 1.2 < qty ≤ 1.65 → 1.65货位,
qty > 1.65 → blocked) yields 5 boundary tests.

### Step 2.2 — Identify Data Variation Dimensions

For each requirement, ask: **"What different data conditions could change the outcome?"**

Map each requirement to the data variations identified in Phase 1 Step 1.4:

```markdown
## Data Variation Dimensions

| Requirement | Data Entity | Variations Needed | Source |
|-------------|------------|-------------------|--------|
| F001 (Credit check) | Customer | Normal credit, Zero credit, No record | test_data_inventory.md §Customers |
| F002 (Stock check) | Material | Sufficient stock, Boundary (80%), Insufficient, Zero stock | test_data_inventory.md §Materials |
| F003 (Approval) | Order Amount | Below threshold, At threshold, Above threshold | Requirement analysis R003 |
```

This ensures the coverage matrix accounts for data-driven scenarios, not just
functional scenarios.

### Step 2.3 — Assign Test Priority

Use this priority framework:

- **P0 (Smoke/冒烟)**: Core path — page load, navigation, critical CRUD. Must pass for
  every build. If a P0 fails, the build is blocked.
- **P1 (Functional/功能)**: Main business flows — data CRUD, search/filter, status
  transitions, calculations. Covers happy paths and common workflows.
- **P2 (Boundary/边界)**: Edge cases — input validation, error handling, boundary
  values, empty states, permission checks, concurrent access.
- **P3 (Integration/集成)**: Cross-module workflows — end-to-end scenarios spanning
  2+ modules, data consistency across modules, long-running processes.

### Step 2.4 — Output: `test_coverage_matrix.md`

```markdown

## Coverage Summary
| Requirement | P0 | P1 | P2 | P3 | Total | Coverage % |
|------------|----|----|----|----|-------|-----------|

## Data Variation Coverage
| Data Entity | Variations Covered | Tests | Coverage % |
|-------------|-------------------|-------|-----------|
| Customer credit profile | 3/3 (Normal, Zero, None) | TC-002, TC-007, TC-012 | 100% |

|---------|---------------|----------|----------|---------------|--------|
```

## Phase 3: Data-Driven Test Cases

### Step 3.1 — Generate Detailed Test Cases

For each scenario in the coverage matrix, write a detailed test case. Each test case
must be executable: a tester should be able to follow the steps without additional
knowledge.

### Step 3.2 — Test Case Format

**Standard (single data point) test case:**

```markdown
| ID | Test Case | Precondition | Steps | Expected Result | Test Data | Module |
|----|-----------|-------------|-------|-----------------|-----------|--------|
```

- **ID**: Unique identifier (TC-001, TC-002...)
- **Test Case**: One-line description of what is being tested
- **Precondition**: Required system state before execution
- **Steps**: Numbered, actionable steps. Be specific about what to click, type, or select.
- **Expected Result**: Observable outcome. "Page displays confirmation message '保存成功'"
- **Test Data**: Specific values to use. Reference the data inventory.
- **Module**: ERP module this test belongs to

**Data-driven (parameterized) test case:**

When the same test logic applies to multiple data variations, use a
data-driven format — one test case row, with a companion data table:

```markdown
| ID | Test Case | Precondition | Steps | Expected Result | Data Table Ref |
|----|-----------|-------------|-------|-----------------|---------------|
| TC-005 | Validate credit limit for various customer profiles | Logged in as 销售员 | 1. Navigate to 销售→销售订单<br>2. Select customer from data table<br>3. Enter order amount from data table<br>4. Click 保存 | See data table | Table-D01 |
```

Then provide the data table with varying inputs and expected outcomes:

```markdown
### Data Table D01: Credit Limit Validation Dataset

| # | Customer | Credit Limit | Order Amount | Expected | Rationale |
|---|----------|-------------|-------------|----------|-----------|
| D01-1 | 品牌A (正常信用) | ¥500,000 | ¥300,000 | 保存成功 | Normal: amount < limit |
| D01-2 | 品牌B (正常信用) | ¥500,000 | ¥500,000 | 保存成功 | Boundary: amount = limit |
| D01-3 | 品牌C (零信用) | ¥0 | ¥1,000 | 拦截提示："信用额度不足" | Zero credit: any amount blocked |
| D01-4 | 品牌D (新客户) | (无记录) | ¥10,000 | 按默认规则处理 | No credit record |
| D01-5 | 品牌E (超额) | ¥100,000 | ¥150,000 | 拦截提示："超出信用额度¥50,000" | Over limit: show deficit |
```

## Phase 4: Data-Driven Automation Scripts

### Step 4.1 — Generate Playwright Script (Parametrized)

Generate a Playwright (Python) project that uses `pytest.mark.parametrize` to run
the same test logic against multiple datasets. Use the `sync_api` variant.

### Step 4.2 — Script Requirements

- Use `pytest` as the test runner with `playwright` plugin
- **Data-driven tests use `@pytest.mark.parametrize`** reading from external data files
- Standard (non-parametrized) tests become `def test_*()` functions
- Include fixtures for login and common setup
- Use `expect()` assertions matching the expected results
- Organize with `pytest.mark` for priority levels: `@pytest.mark.p0`, `@pytest.mark.p1`
- Include `conftest.py` with browser, auth, and data-loading fixtures
- Use data-testid or CSS selectors based on the page structure in `module_texts.txt`
- Include `README.md` with setup and run instructions

### Step 4.3 — Data Loading Patterns

**Pattern A: JSON data loader (conftest.py fixture)**

```python
# conftest.py
import json
import pytest
from pathlib import Path

DATA_DIR = Path(__file__).parent / "test_data"

@pytest.fixture(scope="session")
def test_datasets():
    """Load all test datasets from JSON files."""
    datasets = {}
    for json_file in DATA_DIR.glob("*.json"):
        with open(json_file, encoding="utf-8") as f:
            data = json.load(f)
            for key, value in data.items():
                datasets[key] = value["dataset"]
    return datasets

def load_dataset(name: str) -> list[dict]:
    """Load a specific dataset by name."""
    # Looks for the dataset in test_data/*.json
    for json_file in DATA_DIR.glob("*.json"):
        with open(json_file, encoding="utf-8") as f:
            data = json.load(f)
            if name in data:
                return data[name]["dataset"]
    raise FileNotFoundError(f"Dataset '{name}' not found")
```

**Pattern B: Parametrized test using external data**

```python
# test_functional.py
import pytest
from playwright.sync_api import Page, expect
from conftest import load_dataset
from utils.test_data_loader import get_parametrize_args

# Load data once at module level
CREDIT_DATA = load_dataset("credit_limit_validation")
# Transform into pytest parametrize format: [(id1, customer1, limit1, amount1, expected1), ...]
CREDIT_PARAMS = [
    (d["id"], d["customer"], d["credit_limit"], d["order_amount"],
     d["expected_result"], d.get("expected_message", ""))
    for d in CREDIT_DATA
]

@pytest.mark.p1
@pytest.mark.parametrize(
    "data_id, customer, credit_limit, order_amount, expected, message",
    CREDIT_PARAMS,
    ids=[p[0] for p in CREDIT_PARAMS]  # Use data IDs as test case names
)
def test_credit_limit_validation(
    page: Page,
    login_as_sales,
    data_id: str,
    customer: str,
    credit_limit: float,
    order_amount: float,
    expected: str,
    message: str,
):
    """TC-005: Credit limit validation — data-driven across customer profiles."""
    # Navigate
    page.click("text=销售")
    page.click("text=销售订单")
    page.click("text=新建")

    # Select customer
    page.locator("[data-testid='customer-select']").click()
    page.locator(f"text={customer}").click()

    # Enter order amount
    page.locator("[data-testid='order-amount']").fill(str(order_amount))

    # Save
    page.click("text=保存")

    # Assert based on expected result
    if expected == "pass":
        expect(page.locator(".success-message")).to_be_visible()
    elif expected == "block":
        expect(page.locator(".error-message")).to_contain_text(message)
    elif expected == "use_default":
        expect(page.locator(".warning-message")).to_be_visible()
```

**Pattern C: CSV data loading utility**

```python
# utils/test_data_loader.py
import csv
from pathlib import Path
from typing import Any

DATA_DIR = Path(__file__).parent.parent / "test_data"

def load_csv(filename: str) -> list[dict[str, str]]:
    """Load test data from a CSV file in test_data/ directory."""
    filepath = DATA_DIR / filename
    if not filepath.exists():
        raise FileNotFoundError(f"Test data file not found: {filepath}")
    with open(filepath, encoding="utf-8-sig") as f:  # utf-8-sig handles BOM
        return list(csv.DictReader(f))

def get_parametrize_args(
    filename: str,
    columns: list[str],
) -> list[tuple[Any, ...]]:
    """Load CSV and transform into pytest parametrize argument tuples."""
    rows = load_csv(filename)
    return [tuple(row.get(col, "") for col in columns) for row in rows]
```

## Important Notes

### Data Sources (read in this order)
1. **`references/erp_domain_knowledge.md`** — State machines, business rules, formulas,
   error messages, cross-module flows. Read after SKILL.md for detailed domain context.
2. **`module_texts.txt`** — Live ERP page captures with real entity instances. Ground
   all test data in this file.
3. **Knowledge base documents** (`Documents/知识库文档/`) — Original IMS system docs.
   Reference when deeper module-specific details are needed.

### Test Generation Principles
- **Extract, don't invent** test data — pull real entity names, codes, and values from
  the captured pages. Placeholder data like `<test_customer>` defeats the purpose.
- **Every state transition is a test case.** Use the state machines in the domain
  knowledge reference. Valid transitions → P1, invalid transition attempts → P2.
- **Every formula threshold is a boundary test.** Test at exact threshold, threshold±1,
  zero, and extreme values. Use actual formulas from the domain knowledge reference.
- **Every error message is an expected result.** Use the exact Chinese error messages
  from the domain knowledge reference as assertions.
- Use real ERP page names, field labels, and menu paths from `module_texts.txt`.
- **Data variation is the key to ERP test quality** — a test that passes for one
  customer but fails for another is a real bug. Always identify the data dimensions
  that matter and generate test data covering each variation category.
- For cross-module workflows (P3), trace the data flow through all relevant modules
  using the 10 documented flows in the domain knowledge reference.
- **Account for device-specific behavior** — the same operation may behave differently
  on PC (management), PAD (shop floor), PDA (scanning), and Industrial Tablet (station).
- **Ensure P3 tests are never skipped** — when requirements identify cross-module
  dependencies, P3 integration tests are mandatory, not optional.
- If `module_texts.txt` is not found, ask the user for its location.
- When the user mentions specific modules or pages, prioritize those in the analysis.
